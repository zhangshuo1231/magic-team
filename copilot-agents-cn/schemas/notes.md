# Notes Schema 定义

本文件定义了项目笔记的结构规范。所有涉及 Notes 的 Agent（Conductor、Analyst、Note Keeper）必须严格遵守此结构。

Notes 的目标是：让任何一个阅读它的 Agent，在读完之后，对项目拥有接近"维护了两三年的程序员"那样的全局理解——知道代码在哪、数据怎么流、为什么这样设计、怎么跑起来、哪里有风险。

---

## 〇、设计哲学（请勿删除）

理解本文档的设计意图，对正确维护它至关重要。以下是 Notes 体系的核心设计思路，必须随文档一同保留，不得在更新时丢失。

1. **作者是 AI + 用户，读者是 AI。** Notes 不是给人看的文档，而是给 Agent 读取并据此工作的知识库。因此它的结构服务于 Agent 的检索和判断，而非人的阅读习惯。

2. **知识来源是混合的，因此必须标注可靠性。** 一部分内容由 Analyst 读代码产出（事实性强），一部分是 AI 的推断（"为什么"类问题代码往往无法回答），一部分来自用户或外部材料（设计意图、隐性配置、踩坑经验、Confluence/Jira、diagram、设计文档、聊天摘录、skill 等）。三种标注状态 [已确认]/[推测]/[待补充] 正是为此而设，它们同时是 Conductor 判断"工作条件是否成熟"的信号。

3. **Notes 是一个渐进生长的活知识体，而非一次性快照。** 这是本体系的一个核心特色。对一个 repo 的理解，在现实中总是渐进积累的：Analyst 首次学习只是打下骨架，此后用户会不断补充——今天读代码读懂一点、明天从同事那里听到一段历史、下周开会得知一个设计背景，或者提供一份新的 Confluence 页面、diagram、设计文档、聊天摘录、只言片语、预先写好的 skill。这些零散认知和外部材料都应当被持续地消化并融入 Notes，让它随时间越来越完备。因此文档必须**随时准备好被定点更新和融入（merge）**——更新者要明确知道新信息属于哪个 section、与已有内容是什么关系，把它无缝织进去，而不是简单 append 到末尾或粘贴原文。

4. **新知识在融入前应尽量经过核实。** Notes 的价值在于可信。用户提供的零散补充或新材料，在融入前应尽量验证：
   - 纯主观意图类信息（如"当年为什么这么设计"）代码中无法核实，如实标注其来源与可靠性即可。
   - 但只要补充中夹带了**关于代码实现的论断**（如"X 是通过 Y 调用 Z 的"），就应倾向于由 Analyst 回到 code base 核实，**不要轻易跳过这一步**。一个常见且危险的情况是：有人凭记忆描述某段实现，但代码其实早已被改过，他并不知情——若不核实，就会把过时的描述当作事实写入。核实的结果还能反过来校准相关意图描述的可信度。
   - 是否需要代码核实由 Conductor 裁量，但默认倾向于核实；仅当信息纯属主观意图、不含任何可查证的实现论断时才跳过。

5. **更新必须经过一致性审查，不可盲目写入。** 如果用户这次的补充与文档中已有的 [已确认] 事实明显冲突，不能直接覆盖。Conductor 应指示 Analyst 进行有限范围的代码阅读核实，或与用户确认，澄清矛盾之后再准确更新。Notes 的可信度依赖于这道审查关卡。

6. **文档自身的演进也要留痕。** 包括本设计哲学在内，结构性的设计决策应当被保留和传承，避免未来维护者丢失"为什么这样设计"的上下文。

## 一、文件位置约定

```
{项目根目录}/.copilot-notes/
    project-notes.md       ← 项目级笔记（描述现状，更新覆盖）
    adr.md                 ← 增量决策笔记（append-only 流水账）
```

## 二、内容标注机制（重要）

Notes 中的每一个内容块，都必须标注其信息可靠性状态。这不仅是为了诚实记录，更是为了让 Conductor 据此判断"某项工作的条件是否成熟"。

三种标注状态：

- **[已确认]**：基于代码事实、配置、测试、用户明确确认或已验证的外部文档。可信赖。
- **[推测]**：Analyst 基于代码的合理推断，但无法确认真实意图（尤其是"为什么"类问题）。需谨慎对待，不可当作事实。
- **[待补充]**：代码中无法得知，明确缺失。需用户或外部材料（Confluence/Jira、diagram、设计文档、skill 等）补充。

标注写在每个条目或段落的开头，例如：

```
- [已确认] 系统使用 Spring Boot 3.2，见 pom.xml。
- [推测] 模块间使用 Kafka 通信，推测目的是解耦与异步处理。实际设计意图需用户确认。
- [待补充] 本地启动所需的环境变量清单，代码中未完整记录，需用户提供。
```

**Conductor 使用标注的方式**：当某项 task 涉及的关键信息大量处于 [推测] 或 [待补充] 状态时，Conductor 应判断"条件不成熟"，先向用户追问或安排 Analyst 深入调查，待信息升级为 [已确认] 后再委派实现工作。

**关于下文模板中的 `[标注]`**：在第三节 project-notes.md 的结构模板里，每个条目前出现的 `[标注]` 只是一个**占位符**，表示"此处应填入 [已确认]/[推测]/[待补充] 三者之一"。实际写入 Notes 时，必须替换成真实的标注，绝不能原样保留字面的 `[标注]` 字样。

## 三、project-notes.md 结构

```markdown
# 项目笔记

## Metadata
- last_updated_at: YYYY-MM-DD HH:mm
- source_branch: {生成/更新 Notes 时读取的分支名}
- source_commit_id: {生成/更新 Notes 时作为代码事实依据的 commit ID，判断 Notes 时效性的关键依据}
- notes_scope: {whole-project / module:{模块名} / limited:{范围说明}}
- workspace_state: {clean / dirty / unknown；若 dirty，应简述相关未提交改动}
- author: {更新者的用户ID}

## 1. 项目概述
[标注] 项目是什么、解决什么问题、在更大的系统中处于什么位置。

## 2. 技术栈与版本
[标注] 主要技术栈，具体到版本号（如 Java 17、Spring Boot 3.2、Maven、Kafka 3.x）。
如果能从代码看出选型原因则注明，否则标注 [推测] 或 [待补充]。

## 3. 代码地图
本节是 Planner 快速定位代码的核心依据。
- [标注] 目录结构说明：每个顶层目录/包的职责
- [标注] 关键入口点：主要功能的入口类和方法在哪
- [标注] "我想改 X，应该去看哪些文件" 的对应关系

## 4. 模块划分与职责
[标注] 每个模块做什么、模块之间的依赖关系。

## 5. 核心数据流
[标注] 端到端追踪几条最重要的链路：
请求/事件/任务从哪里进来（必要时包括跨服务入口）→ 经过哪些层/类 → 数据如何流转 → 最终在哪里落地。

## 6. 关键设计决策
本节大概率包含较多 [推测] 和 [待补充]，因为代码通常无法说明"为什么"。
- [标注] 重要的架构选择及其原因（为什么这样而不那样）
- [标注] 哪些是历史遗留、不要轻易改动的部分

## 7. 数据模型
[标注] 核心的数据结构、表结构、关键实体及其关系。

## 8. 外部依赖与集成
[标注] 外部系统与内部集成关系，包括相关服务/微服务、数据库/共享 schema、消息队列/topic、第三方服务、API，以及集成方式。

## 9. 构建与运行
本节支撑本地调试与验证场景。
- [标注] 如何 build、如何本地启动、如何调试
- [标注] 关键配置项和环境变量（隐性的本地配置往往需 [待补充]）

## 10. 测试策略
[标注] 测试如何组织、如何运行、覆盖情况如何。

## 11. 风险区与注意事项
本节多为经验性知识，代码可识别高耦合/高复杂度区域，但真实"踩坑经验"需用户补充。
- [标注] 哪些地方脆弱、改动需谨慎
- [标注] 已知的坑、约定、反直觉的设计
```

## 四、adr.md 结构

```markdown
# 增量决策笔记（ADR）

最新的记录永远在最上方。旧记录不删除。

---

## {YYYY-MM-DD} | {Jira Ticket ID} | {一句话描述这次改动}

### Metadata
- branch: {feature branch 名称}
- base_commit_id: {本次改动基于的 base commit ID，通常是目标分支与 feature branch 的 merge-base}
- working_tree_state: {uncommitted / pending-commit / committed}
- diff_summary: {本次 ADR 对应的工作区改动摘要或涉及文件列表}
- final_commit_id: {本次改动最终 commit 的 ID；若尚未提交，填写 pending}
- author: {用户ID}

### 改动范围
{列出本次修改了哪些模块、文件}

### 改动原因
{为什么要做这个改动，对应的 task 目标}

### 技术决策
{做了哪些重要技术决策，以及为什么这样决定而不是其他方案}

### 风险与注意事项
{本次改动引入了哪些风险，或后续需要注意什么}

---

{下一条 ADR 记录}
```

## 五、Conductor 判断 Notes 完备性的规则

Conductor 读取 `project-notes.md` 的 Metadata 后，按以下逻辑判断：

1. **Notes 不存在** → 必须调用 Analyst 从头学习。
2. **source_commit_id 是当前目标分支或当前 HEAD 的祖先 / 等于当前 HEAD，且 notes_scope 覆盖 task 相关范围** → Notes 基本有效。Conductor 自行查看从 source_commit_id 到当前工作区的增量 diff：
   - 增量小且不涉及 task 相关模块 → 直接进行，无需调用 Analyst。
   - 增量较大或涉及 task 相关模块 → 告知用户，考虑调用 Analyst 做增量更新。
3. **source_commit_id 缺失、不是当前代码历史的祖先，或 notes_scope 不覆盖 task 相关范围** → 视为过期或适用性不足，告知用户，建议调用 Analyst 重新学习或做有限范围调查。
4. **workspace_state 为 dirty 或 unknown** → 不直接判定 Notes 无效，但涉及当时未提交改动的区域时要降低置信度；若该区域与当前 task 相关，应安排 Analyst 重新核实。

此外，即使时效性 OK，Conductor 还需检查 task 相关内容的**标注状态**：若关键信息多为 [推测]/[待补充]，则判断条件不成熟，先补全信息。

## 六、定点更新与一致性审查协议

用户可能在任何时候——不一定伴随某个具体 task——单独要求补充或修正 Notes，也可能提供一份新文档、diagram、Confluence 页面、截图/PDF、聊天摘录、只言片语、预先写好的 skill 或其他 artifact 来完善知识库。这正是 Notes 作为"渐进生长的活知识体"的日常体现。处理这类更新时，必须遵循以下流程，而不是简单地把内容追加到文档末尾或把原始材料全文粘贴进 Notes。

**第一步：判断材料性质、可读性与是否需要代码核实（Conductor 裁量）。**
先判断材料来源、目标项目、格式和可访问性；如果材料无法读取、链接无权限、diagram 信息不足或上下文缺失，应先要求补充可读内容。随后检查用户补充或材料中是否夹带了**关于代码实现的论断**：
- 含可在 code base 中查证的实现论断 → 倾向于指派 Analyst 做核实型调查（不要轻易跳过）。Analyst 会先读现有 Notes 的相关部分，再带着该论断到代码中查找证据，报告核实结果（证实 / 证伪 / 部分证实 / 代码中无法判定）。
- 纯主观意图类信息（代码中无法核实）→ 跳过核实，但要如实标注其来源与可靠性。
- 注意：意图与实现常被一起陈述。说话人对意图可能真诚，但对实现现状的记忆可能已过时（代码被改过而他不知情）。因此只要含实现论断就应核实，核实结果也可反过来校准相关意图描述的可信度。
- 对大型、混合或格式不确定的材料 → 倾向于先指派 Analyst 做文档消化型调查，抽取候选知识、分类、识别需代码核实的部分，再交由 Note Keeper 融入。

**第二步：定位。**
判断信息（及其核实结果）属于 project-notes.md 的哪一个 section（项目概述、代码地图、核心数据流、关键设计决策……）。更新必须写入对应 section。如果信息横跨多个 section，分别写入相应位置。

**第三步：融入（merge），而非追加。**
这是关键——新信息要被**无缝织入**文档的合适位置，而不是堆在末尾，也不是把原始文档全文塞进 Notes。判断新信息与已有内容的关系，采取相应的融入方式：
- **填补**：补上某个 [待补充] 的空白。
- **深化/扩展**：在已有条目基础上增加细节，使其更完整。
- **修正**：纠正某个 [推测] 或已过时的 [已确认]（须先通过下面的一致性审查）。
- **关联**：若信息同时影响多个 section，在各处分别融入并保持彼此一致。
融入后，文档应保持连贯、不冗余，读起来仍是一个整体，而不是拼接的碎片。

**第四步：一致性审查。**
将新信息与文档中已有内容比对：
- 若填补 [待补充] 空缺，或与已有内容一致 → 直接融入，并视情况（尤其经代码核实证实后）将标注升级为 [已确认]。
- 若与已有的 [推测] 冲突 → 用户信息（或核实结果）优先，更新内容并相应调整标注。
- 若与已有的 **[已确认]** 事实明显冲突 → **不可直接覆盖**。Conductor 应：指示 Analyst 做代码核实，或向用户指出矛盾请求澄清；矛盾解决后再准确更新。

**第五步：更新 Metadata。**
任何写入完成后，由 Note Keeper 同步更新对应文件的 Metadata。

这套流程是 Notes 可信度的保障：Notes 之所以能被 Conductor 当作判断依据，正是因为写入它的内容尽量都经过了核实并被妥善融入，而非随意堆积。

## 七、重要约定

- Notes 是跨 session 持久化的唯一结构化数据。
- project-notes.md 描述"现状"，内容随项目演进更新覆盖。
- adr.md 是 append-only 的流水账，新记录永远加在最上方，旧记录不删除。
- ADR 可以在改动尚未 commit 时记录，`final_commit_id` 此时填写 `pending`，待用户完成提交后可由 Note Keeper 回填或追加说明。
- 每次 Note Keeper 更新任何一个文件后，必须同步更新对应文件的 Metadata。
- 所有内容块必须带 [已确认]/[推测]/[待补充] 标注，Analyst 产出时不得伪装确定性。
