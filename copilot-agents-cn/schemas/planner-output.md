# Planner Output Schema 定义

本文件定义 Planner 产出的"实现计划"的标准格式。

**上游**：Conductor 向 Planner 提供结构化任务包。
**下游消费者**：
- Coder —— 照着这份计划进行实现。
- Verifier —— 照着这份计划验证实现是否符合预期。

因此本格式必须做到：让 Coder 不需要再揣摩意图就能动手，让 Verifier 有明确的对照标准。

---

## 一、设计原则

1. **计划而非代码。** Planner 描述"要做什么、在哪里做、怎么做",但不写出完整的实现代码。具体编码是 Coder 的职责。Planner 可以给出关键的代码片段示意或方法签名,但不替 Coder 完成实现。

2. **可对照。** 每一个改动点都应清晰到 Verifier 可以逐条核对。

3. **诚实标注不确定性。** 如果计划中某些部分依赖尚未确认的假设,必须明确标出,而不是假装一切都已确定。

---

## 二、输出结构

````markdown
# 实现计划

## 0. 任务概要
- Jira Ticket: {ticket ID 和标题}
- 任务类型: {新 feature / bug fix / 逻辑修改 / refactor / CI-CD / 文档 / 其他}
- 一句话目标: {用一句话说明这次要达成什么}

## 1. 背景与理解
{Planner 对这个 task 的理解。包括从任务包和相关代码中获取的关键背景。
说明这次改动要解决的问题,以及为什么需要这样做。}

## 2. 影响范围
列出本次改动会涉及的所有位置:
- 文件 / 类 / 方法: {具体路径和名称}
- 每一处的改动性质: {新增 / 修改 / 删除}
- 影响的模块和潜在的连锁影响

## 3. 实现步骤
按顺序列出 Coder 应执行的步骤。每一步包括:
- 步骤序号与简述
- 涉及的文件 / 类 / 方法
- 该步骤要做的具体改动描述
- (可选)关键的方法签名、数据结构或代码片段示意

步骤应足够具体,使 Coder 无需重新猜测意图。

## 4. 遵循的既有约定
说明本次实现必须遵循的项目既有规范:
- 代码风格、命名约定
- 既有的框架用法、设计模式
- 必须复用的既有组件 / 工具类
(这些信息应来自 Notes 或任务包)

## 5. 假设与待确认项
- [假设] {计划所基于的、尚未被确认的假设}
- [待确认] {需要用户或后续核实才能最终确定的点}
如果本节非空,说明计划存在不确定性,Coder 实现前 Conductor 可能需要先澄清。

## 6. 测试关注点
为 Test Writer 提供方向(非强制由 Planner 写测试,只是提示):
- 需要重点覆盖的正常路径
- 关键的边界条件
- 可能的异常场景

## 7. 风险与注意事项
- 本次改动可能引入的风险
- 需要特别小心的地方
- 对其他模块或功能的潜在副作用
````

---

## 三、Planner feedback 机制

如果 Planner 在分析任务包和代码后,认为**信息不足以制定可靠的计划**,它不应勉强输出计划,而应改为输出一份 feedback,返回给 Conductor。feedback 格式:

````markdown
# 信息不足反馈

## 无法制定可靠计划的原因
{说明缺少什么关键信息}

## 需要补充的内容
- {具体列出需要 Conductor 向用户追问、或需要 Analyst 补充调查的内容}

## 当前已掌握的部分
{说明在现有信息下,已经能确定的部分,避免重复劳动}
````

Conductor 收到 feedback 后,负责向用户追问或安排 Analyst 调查,补全后再次委派 Planner。

---

## 四、重要约定

- Planner 是只读的,本格式产出的是计划文本,不涉及任何文件修改。
- 计划的详细程度以"Coder 无需再猜测意图"为标准。
- 任何不确定性都必须显式标注在第 5 节,不得隐藏。
- 当信息不足时,输出 feedback 而非勉强的计划。
